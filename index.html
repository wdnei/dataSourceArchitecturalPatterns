Introdução

O coração de todo aplicativo é o acesso a dados. Desenvolvedores passam muito de seu tempo projetando, avaliando, testando e se preocupando com as formas de acesso a dados em seus sistemas. Assim, é eminente o surgimento de padrões comuns para escrever código da camada de acesso a dados.

O padrões de acesso a dados apresentados aqui foram definidos por Martin Fowler, são eles:

*Table Data Gateway *Row Data Gateway *Active Record *Data Wrapper

Table Data Gateway

Este Padrão é também denominado como Data Access Object(DAO) Um objeto que age como um Gateway(Ponte/Porta de Entrada) as tabelas do banco de dados. Table Data Gateway inclui todos comando SQL para acessar uma única tabela ou view: selects, inserts, updates, and deletes. As propriedades são implementadas de acordo com a necessidade.

Porque usar:

-Encapsular os comando SQL e manter o acesso ao banco mais transparente -Evitar repetição de codigo

Quando Usar:

-Table Data Gateway trabalha perfeitamente com Table Module, onde produz um DataSet para trabalhar com as tabelas do banco em memória. -Quando há o uso de Transaction Script.

Row Data Gateway

Row Data Gateway representa objetos exatamente como as linhas das tabelas do banco de dados , mas que podem ser acessados por mecanismos em linguagens de programação. Todos os detalhes de acesso ao banco são escondidas usando interfaces. Problema com múltiplas instancias do mesmo registro. Necessita implementar todas as propriedades de acesso.

Porque usar:

-Evitar repetição de código. -Quando não há o uso de classes de domínio.

Quando Usar:

-Quando há o uso de Transaction Scripts.

Active Record

Uma classe contem as propriedades da tabela, encapsula o código de acesso(SQL) ao banco de dados, e também contém a lógica de negócio, ou seja, um objeto contém ambos dados e comportamento. Entretanto, há problemas com múltiplas instancias do mesmo registro.

Porque/Quando Usar:

-Boa escolha quando há o uso de Domain Model -Quando há a necessidade de mesclar acesso a dados e a lógica de negócio -Quando não há muita complexidade: herança, relação entre objetos.

Data Wrapper

Uma camada que mapeia e move dados entre objetos e banco de dados, os mantendo independentes. Busca um registro no banco de dados e cria um objeto com as propriedades preenchidas. Onde há o isolamento do acesso ao banco de dados e da lógica de négocio. Podendo usar Lazy Load para resolver relacionamento de dados.

Abordagens de implementação:

-Uso de ferramentas para criar classes a partir do mapeamento do banco. -Uso de Metadata e Reflection
